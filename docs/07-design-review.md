# مراجعة التصميم الشاملة (قبل اعتماد المالك)

**الحالة:** جاهز للاعتماد | **يراجع:** المستندات 01 إلى 06 مجتمعة

هذه مراجعة تقاطعية (Cross-Document Review) بدور **Technical Reviewer**: لا
تراجع كل مستند بمعزل عن غيره، بل تبحث تحديدًا عن تناقض بين مستندين. أي
خلل وُجد إما أُصلح فورًا (ومُوثَّق أدناه)، أو أُجِّل صراحة كبند عمل لاحق
غير حاجب لاعتماد التصميم.

## ما تم فحصه بنجاح (لا تناقض)

| الفحص | النتيجة |
|---|---|
| تسميات الحالات عبر الطبقات (قاعدة البيانات ← API ← الواجهة) | `open/in_progress/resolved/closed` للتذاكر، `low/medium/high/urgent` للأولوية، `high/medium/low` لثقة الذكاء الاصطناعي — نفس القيم حرفيًا في `01-database-design.md`، `06-api-design.md`، والعرض التجريبي |
| الأدوار (RBAC) | `owner/store_manager/agent` متطابقة بين جدول `roles` وجدول الصلاحيات في المعمارية وتدفق تسجيل الدخول §1 |
| شرط العزل عبر الطبقات الثلاث | التطبيق + RLS (قاعدة البيانات) + فلتر `store_id` في استعلام RAG — والآن أيضًا **حد شبكي رابع لم يكن مذكورًا صراحة من قبل**: نقطة `/v1/internal/ai/query` في عقد API غير مكشوفة خارج الشبكة الداخلية أصلًا، فلا عميل خارجي يستطيع استدعاءها بمعزل عن سياق متجر محدد — طبقة حماية إضافية تُذكر الآن في هذا السجل |
| مسارات الـwebhook الموحدة | مسار قناة واحد `/webhooks/channels/{channelTypeKey}/{channelAccountId}` ومسار تكامل واحد `/webhooks/integrations/{platformKey}/{integrationId}` — يطابقان فعليًا بند "إضافة قناة/تكامل دون تعديل جوهري" في المتطلبات الأصلية |
| خريطة الشاشات | كل شاشة في `04-user-flows.md` لها تطبيق فعلي في العرض التجريبي (`05-ui-screens.md`) — لا شاشة موثّقة بلا تنفيذ، ولا شاشة منفَّذة بلا تدفق يبرر وجودها |

## تناقضات وُجدت وأُصلحت

### 1. تفاصيل التذكرة كانت منقوصة في العرض التجريبي

**المشكلة:** حقول التذكرة الإلزامية حسب قاعدة البيانات والمتطلبات الأصلية
(بيانات العميل، ملخص المحادثة، سبب التصعيد، المرفقات، توصية الذكاء
الاصطناعي) موجودة كأعمدة في `tickets`، لكن لوحة التذاكر في العرض التجريبي
كانت تُظهر فقط العنوان والأولوية والقسم — بقية الحقول غير مرئية لأي مستخدم.

**الإصلاح:** أُضيفت لوحة جانبية (Side Panel) فعلية تُفتح بالنقر على أي
تذكرة، تعرض كل الحقول المطلوبة كاملة (بما فيها القسم والأولوية كحقلين
قابلين للتعديل مباشرة)، مطابقة تمامًا لمكوّن "اللوحة الجانبية" الموصوف في
نظام التصميم.

### 2. وصف مكوّن "اللوحة الجانبية" كان يخلط بين حالتين مختلفتين

**المشكلة:** نظام التصميم وصف اللوحة الجانبية كحل لعرض "تفاصيل تذكرة" *و*
"تفاصيل محادثة" معًا، بينما صندوق الوارد الفعلي (والتدفق الموثَّق له) يعرض
تفاصيل المحادثة كعرض دائم مقسوم (Split View)، لا كطبقة عائمة — لأن صندوق
الوارد هو شاشة العمل الأساسية اليومية للموظف، وإغلاقها/فتحها باستمرار
احتكاك غير مبرر.

**الإصلاح:** عُدِّل نص `03-design-system.md §5` ليحصر اللوحة الجانبية
بالسياقات الثانوية (كتفاصيل تذكرة من قائمة أو لوحة)، ويوضّح صراحة أن
صندوق الوارد استثناء مقصود بعرض مقسوم دائم.

## بنود مؤجَّلة (لا تحجب الاعتماد)

هذه فجوات حقيقية لكنها بصرية/إضافية بحتة، بلا أي أثر معماري أو على شرط
العزل — تُترك لمرحلة بناء الواجهة الأمامية الفعلية بدل تأخير الاعتماد:

| البند | لماذا لا يحجب الاعتماد |
|---|---|
| حالات "التنبيه" (Toast) وحالة "فارغة" (Empty State) موصوفتان في نظام التصميم لكن غير مُجسَّدتين حيًا في العرض التجريبي | مكوّنان بصريان مستقلان، لا تصميم إضافي مطلوب، تُبنى مباشرة حسب `03-design-system.md §5` وقت البرمجة |
| لا شاشة لإدارة أقسام التذاكر (`ticket_departments`) رغم وجود نقطتها في عقد API | إضافة بسيطة (نموذج + جدول) داخل شاشة الإعدادات الحالية، لا تتطلب تصميم تدفق جديد |
| تغيير حالة التذكرة (سحب بين أعمدة الكانبان) غير مُفعَّل تفاعليًا في العرض التجريبي | العرض التجريبي أداة توضيح تصميم لا تطبيق يعمل، وآلة الحالة موثَّقة بالكامل نصيًا في `02-architecture.md §6` |

## الخلاصة

لا يوجد أي تعارض بين قاعدة البيانات والمعمارية ونظام التصميم والتدفقات
وعقد API يمنع المتابعة. التصميم **جاهز لاعتماد المالك** بصيغته الحالية.
بعد الاعتماد، الخطوة التالية مباشرة: بناء Backend حسب `06-api-design.md`
موديولاً موديولاً بالترتيب في `02-architecture.md §12`.
