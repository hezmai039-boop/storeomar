## 0. الغرض والنطاق

هذا الفصل يوثّق موديولاً إضافيًا مستقلاً بالكامل (`backend/src/modules/ai-intelligence/`)
يمنح كل متجر فريق وكلاء ذكاء اصطناعي متخصصين، بأدوات (Tools) تصل لبيانات
حية بدل الاعتماد على قاعدة المعرفة فقط، وذاكرة متعددة المستويات، وبحثًا
هجينًا، وسجل تدقيق كامل لكل استدعاء أداة وكل قرار توجيه.

**قيد صارم غير قابل للتفاوض طُبِّق حرفيًا:** لا تعديل على أي جدول أو
واجهة برمجية أو تدفق قائم فعليًا. كل شيء هنا إضافة صرفة:

- **6 جداول جديدة فقط** (`ai_specialists`, `ai_tool_invocations`,
  `ai_orchestrator_runs`, `ai_customer_memory`, `ai_conversation_memory`,
  `ai_business_memory`) — لا عمود واحد أُضيف أو عُدِّل على أي جدول قائم.
- **مسار الإنتاج الحي غير مُلموس إطلاقًا:**
  `src/modules/channels/webhook.ts` (الذي يستدعي `gatherAiContext`/
  `completeAiPipeline` من `aiPipeline.ts` لكل رسالة عميل حقيقية) لم يُعدَّل
  بحرف واحد. الطبقة الجديدة تُعرض حصرًا عبر مسارات API جديدة تحت
  `/v1/stores/:storeId/ai-intelligence/*`، محمية بنفس RBAC الحالي.
- **6 أسطر فقط أُضيفت لملفات موجودة** خارج `schema.prisma`/`rls.sql`
  (سطر تصدير `tokenize` في `retrieval.ts`، سطرا صلاحيات في
  `permissions.ts`، سطرا استيراد/تركيب في `index.ts`) — كل سطر إضافة صرفة
  لا تُغيّر سلوكًا قائمًا؛ تحقّق ذلك بـ `git diff --stat` بعد هذا الفصل.

**ما هذه المرحلة ليست إياه، بصراحة:** ليست 17 استدعاء LLM مستقلاً يعمل
لكل رسالة (تكلفة وزمن استجابة غير مبرَّرين) — بل مُنسِّق واحد (Orchestrator)
يصنّف النية ويوجّه لشخصية متخصصة واحدة تستخدم أدوات حقيقية عبر استدعاء
دوال (Tool-Calling) فعلي مع Claude، بمعمارية Router + Specialists الموصى
بها أصلاً في [13-ai-engine-architecture.md §3.8](13-ai-engine-architecture.md#38-نموذج-تعاون-الوكلاء-المتعددين).
هذا قرار هندسي واعٍ، لا اختصارًا.

## 1. المعمارية

```
POST /v1/stores/:storeId/ai-intelligence/ask
        │
        ▼
gatherOrchestratorContext (DB فقط، معاملة قصيرة)
  ├─ ensureDefaultSpecialists  → يزرع 17 شخصية افتراضية لأول مرة فقط
  ├─ classifyIntent            → تصنيف كلمات مفتاحية v1 (بلا شبكة/تكلفة)
  ├─ refreshConversationMemory → ملخص متجدد لآخر 12 رسالة
  └─ hybridSearchKnowledge     → بحث كلمات + Full-Text Search مدمَجان
        │
        ▼
completeOrchestratorRun (شبكة، بلا معاملة قاعدة بيانات مفتوحة)
  └─ runAgentWithTools          → محادثة Tool-Use حقيقية مع Claude
        └─ لكل استدعاء أداة: معاملة قصيرة مستقلة عبر withStoreContext
        │
        ▼
تسجيل AiOrchestratorRun + AiToolInvocation (معاملات قصيرة)
```

### الوحدات

| الملف | الدور |
|---|---|
| `tenantGuard.ts` | نسخة ثالثة مستقلة من فحص UUID — ترفض أي عملية بلا `store_id` صالح، دفاع متعمّق فوق RLS |
| `specialists.ts` | تعريف 17 شخصية افتراضية + زرعها الكسول (Lazy) لكل متجر عند أول استخدام |
| `classifier.ts` | تصنيف نية v1 بكلمات مفتاحية عربية — منفذ (seam) موثّق للترقية لاحقًا |
| `tools/*` | 10 أدوات حقيقية (لا وهمية) تُبنى فوق الجداول القائمة فعليًا (`synced_products`, `synced_orders`, `customers`, `tickets`, `knowledge_chunks`, `store_daily_metrics`) |
| `hybridSearch.ts` | دمج بحث الكلمات المفتاحية القائم + Postgres Full-Text Search |
| `memory.ts` | ذاكرة محادثة قصيرة + ذاكرة عميل طويلة + ذاكرة عمل، كلها بـ`store_id` |
| `agentRuntime.ts` | حلقة Tool-Use فعلية مع Anthropic Messages API (لا محاكاة) |
| `orchestrator.ts` | يربط كل ما سبق، بنفس انقسام DB/شبكة المعتمد في `aiPipeline.ts` |
| `routes.ts` | 8 مسارات API جديدة تحت RBAC كامل |

## 2. العزل متعدد المستأجرين (دفاع متعدد الطبقات)

1. **RLS على مستوى قاعدة البيانات** — الجداول الستة الجديدة أُضيفت لمصفوفة
   `prisma/rls.sql` بلا استثناء، بنفس السياسة المعتمدة لكل جدول آخر.
2. **فلترة صريحة بـ`store_id` في كل استعلام** — لا استثناء واحد، مُثبَت
   آليًا (انظر §4 أدناه، لا ادّعاءً نثريًا).
3. **`requireTenant()` كطبقة ثالثة مستقلة** — كل دالة دخول (tools، memory،
   orchestrator) ترفض فورًا إن غاب `store_id` صالح، حتى قبل وصول
   الاستعلام لقاعدة البيانات.
4. **ربط هوية من السياق المصادَق عليه، لا من مخرجات النموذج.** هذه نقطة
   حرجة اكتُشفت أثناء مراجعة أمنية ذاتية لهذه المرحلة (انظر §5): أداة
   `CreateEscalationTicket` كانت تقبل مبدئيًا `conversationId`/
   `customerId`/`organizationId` كوسائط يُقرّرها النموذج ذاته — ما يفتح
   احتمال إسناد تذكرة لمحادثة/عميل خطأ نتيجة هلوسة أو صياغة ملتبسة من
   العميل. صُحِّح ذلك: هذه الحقول الآن جزء من `ToolContext` الذي يملؤه
   المُنسِّق من سجل المحادثة الحقيقي المصادَق عليه، لا من مخطط JSON الذي
   يراه النموذج إطلاقًا (`tools/types.ts`، `tools/ticketTools.ts`).

## 3. الإثبات الآلي (لا الوصف النثري فقط)

`src/modules/ai-intelligence/__tests__/tenantIsolation.test.ts` يستدعي كل
أداة عبر Prisma وهمي مُسجِّل (Recording Fake، لا قاعدة بيانات حقيقية) ثم
يفحص **كل** استدعاء DB صادر عنها، ويرسب الاختبار فورًا إن غاب `store_id`
من أي `where`/`data`، أو إن اختلف عن المتجر المُمرَّر. هذا يكتشف أي أداة
مستقبلية تُضاف بلا فلترة صحيحة، آليًا وعلى كل تشغيل.

**حدود هذا الإثبات — بصراحة:** هذه اختبارات وحدة (Unit) بلا قاعدة بيانات
حقيقية (بيئة هذه الجلسة بلا Postgres متاح). تثبت أن **كود التطبيق** يمرّر
`store_id` بشكل صحيح دائمًا؛ لا تثبت أن **RLS نفسها** تمنع تجاوزًا فعليًا
عبر SQL خام — ذلك يحتاج اختبار تكامل ضد Postgres حقيقي (نفس القيد المذكور
أصلاً في `docs/14-security-compliance-infrastructure.md §1.3`). خطوة
لاحقة موصى بها: اختبار تكامل يُشغَّل في CI ضد قاعدة بيانات مؤقتة، ينشئ
متجرين ويحاول قراءة بيانات أحدهما بسياق الآخر مباشرة عبر Prisma الحقيقي.

تشغيل الاختبارات: `npm test` (داخل `backend/`) — 32 اختبارًا، جميعها
تمر، بلا أي اعتماد خارجي (لا مفتاح API، لا قاعدة بيانات).

## 4. قرارات هندسية واعية (ولماذا لا غيرها)

| القرار | البديل المرفوض | السبب |
|---|---|---|
| مُنسِّق واحد + شخصيات متخصصة، لا 17 وكيلاً حيًا مستقلاً | استدعاء LLM منفصل لكل "وكيل" مذكور في الطلب | تكلفة وزمن استجابة غير مبرَّرين لسؤال واحد؛ نفس التوصية في docs/13 §3.8 |
| تحويل يدوي مصغّر من zod إلى JSON Schema (`agentRuntime.ts`) بدل مكتبة خارجية | إضافة تبعية `zod-to-json-schema` | 10 مخططات أدوات بأشكال معروفة ومحدودة؛ ~20 سطرًا مُختبرة أبسط من تبعية جديدة لمثل هذا النطاق |
| Full-Text Search بإعداد `'simple'` (بلا تجذير عربي) | تجاهل FTS كليًا، أو ادّعاء دعم عربي كامل غير موجود في Postgres الافتراضي | Postgres لا يشحن قاموس عربي جاهزًا؛ توثيق القيد صراحة أفضل من الصمت عنه أو الادّعاء الخاطئ |
| كل استدعاء أداة يفتح معاملة DB قصيرة مستقلة خاصة به، لا معاملة واحدة تلف كل حلقة Tool-Use | معاملة واحدة طويلة حول كل المحادثة مع النموذج | نفس المنطق الذي وثّقه `aiPipeline.ts` أصلاً لعدم إبقاء معاملة DB مفتوحة عبر استدعاء شبكي — هنا الاستدعاء الشبكي متعدد الجولات لا جولة واحدة، فالمشكلة أشد |
| زرع الشخصيات الافتراضية كسولاً (Lazy) عند أول استخدام لكل متجر | تعديل `prisma/seed.ts` لإضافتها للمتاجر التجريبية الستة | يتجنّب أي تعديل على مسار الزرع القائم فعليًا؛ يعمل تلقائيًا لأي متجر حالي أو مستقبلي بلا خطوة ترحيل إضافية |
| `ai_specialists.allowed_tools` كمصفوفة قابلة للتعديل من الواجهة | ثوابت مبرمَجة لكل شخصية | يحقق حرفيًا "كل Agent يمتلك Tools/Policies" بلا تغيير كود لكل تخصيص متجر |

## 5. ما لم يُنفَّذ عمدًا في هذه المرحلة (لا نقصًا مخفيًا)

- **لا ربط تلقائي بمسار القناة الحي.** تفعيل هذا المُنسِّق كبديل/مكمّل
  لـ`aiPipeline.ts` داخل `webhook.ts` قرار متعمَّد يمس تدفقًا حيًا يعمل
  فعليًا — القيد الصريح في هذه المهمة كان عدم لمس أي تدفق قائم، فتُرك هذا
  كخطوة "تفعيل" منفصلة تحتاج موافقة صريحة لاحقًا، لا كجزء ضمني من هذا
  التسليم.
- **لا بحث متجهات (Vector/Embeddings) فعلي.** العمود `embedding
  vector(1536)` جاهز في `knowledge_chunks` منذ التصميم الأصلي، لكن لا يوجد
  مزوّد Embeddings مُفعَّل في المشروع (Claude لا يعرض نقطة نهاية
  Embeddings). المنفذ (Seam) موثّق بدقة في `hybridSearch.ts` لإضافته لاحقًا
  دون تغيير أي استدعاء آخر.
- **`marketing`/`manager`/`workflow` بلا أدوات فعلية تقريبًا.** لا توجد
  وحدة حملات/كوبونات ولا محرك قواعد قابل للتهيئة في المخطط الحالي (فجوة
  موثّقة أصلاً في `docs/11-product-feature-gap-analysis.md §5` البند 3).
  شخصياتها موجودة الآن (Prompt قابل للتخصيص) لتوسعتها لاحقًا بلا تغيير
  مخطط قاعدة بيانات إضافي.
- **تصنيف النية v1 كلمات مفتاحية، لا تعلّم آلي.** قرار مقصود لصفر تكلفة/
  زمن استجابة إضافي؛ خارطة الترقية موثّقة في `classifier.ts` وفي
  `docs/13-ai-engine-architecture.md §3.5`.
- **لا واجهة Frontend لهذه الطبقة بعد.** هذا التسليم Backend بالكامل —
  المسارات الثمانية جاهزة للاستهلاك، وشاشة "منشئ الأتمتة"/اختبار الوكيل
  المقترحة في `docs/12-ux-ui-benchmark-and-proposal.md §4.3-4.4` تستهلكها
  مباشرة كخطوة تالية منفصلة.

## 6. مرجع API الجديد

كل المسارات تحت `authenticate` + `requireStoreAccess()`، وصلاحيتان جديدتان
(`ai_intelligence.view`, `ai_intelligence.manage`) مُضافتان لمصفوفة الأدوار
القائمة دون كسر أي صلاحية سابقة.

| Method | Path | الصلاحية | الوصف |
|---|---|---|---|
| GET | `/v1/stores/:storeId/ai-intelligence/specialists` | view | قائمة الشخصيات (تُزرع تلقائيًا أول مرة) |
| PATCH | `/v1/stores/:storeId/ai-intelligence/specialists/:key` | manage | تعديل الاسم/الـPrompt/الأدوات المسموحة/التفعيل |
| GET | `/v1/stores/:storeId/ai-intelligence/tools` | view | كتالوج الأدوات المتاحة في الكود |
| POST | `/v1/stores/:storeId/ai-intelligence/ask` | view | تشغيل المُنسِّق على محادثة قائمة (وحدة الاختبار الحي) |
| GET | `/v1/stores/:storeId/ai-intelligence/memory/customers/:customerId` | view | ذاكرة عميل محدد |
| GET | `/v1/stores/:storeId/ai-intelligence/runs` | view | سجل تشغيلات المُنسِّق (Cursor pagination) |
| GET | `/v1/stores/:storeId/ai-intelligence/tool-invocations` | view | سجل تدقيق كل استدعاء أداة |

## 7. التفعيل الفعلي (خطوات تشغيل، لا افتراض)

1. `npx prisma db push` — ينشئ الجداول الستة الجديدة فعليًا (بلا تغيير
   على أي جدول قائم).
2. `npm run apply-rls` — يعيد تطبيق `prisma/rls.sql` كاملاً (بما فيه
   الجداول الجديدة).
3. `npm run prisma:generate` — إن لم يُشغَّل تلقائيًا بعد الخطوة 1.
4. لا حاجة لأي متغير بيئة إضافي — تعمل الطبقة كاملة بأدواتها وبحثها
   الهجين وذاكرتها بلا `ANTHROPIC_API_KEY`؛ فقط حلقة Tool-Use الفعلية مع
   النموذج تُفعَّل تلقائيًا عند وجوده، بنفس فلسفة `src/lib/llm.ts` القائمة.
