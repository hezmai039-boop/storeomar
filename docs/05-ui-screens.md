# واجهات الشاشات — العرض التجريبي (Prototype)

**الحالة:** جاهز للمراجعة | **يعتمد على:** [03-design-system.md](03-design-system.md), [04-user-flows.md](04-user-flows.md)

الشاشات الست المحدَّدة في خريطة [04-user-flows.md](04-user-flows.md#ملخص-خريطة-الشاشات-المطلوبة)
مبنية كعرض تفاعلي واحد (HTML/CSS حي، بدون إطار عمل بعد — هذا نموذج تصميم
وليس كودًا للإنتاج) يطبّق نظام "Atlas" بالكامل: الألوان، الطباعة،
المسافات، والمكوّنات.

**رابط العرض التجريبي:** يُشارَك كـ Artifact منفصل داخل المحادثة (وليس
كملف HTML ثابت هنا) لأنه تفاعلي — التنقل بين الشاشات فعلي بالنقر على
الشريط الجانبي، وليس صورًا ثابتة.

## القرار المعماري للواجهة الأمامية (تمهيدًا للتنفيذ الفعلي)

راجعه **Senior Frontend Engineer** و**Chief Architect** معًا: التطبيق
الفعلي يُبنى كمكوّنات مستقلة قابلة لإعادة الاستخدام (نفس فلسفة الموديولات
في المعمارية الخلفية)، بحيث:

- كل مكوّن من القسم 5 في نظام التصميم = مكوّن برمجي واحد فقط، لا نسخ
  متعددة بأشكال مختلفة لنفس الغرض (زر واحد بمتغيرات، لا 3 مكوّنات أزرار).
- شاشات الـ Workspace (صندوق الوارد، التذاكر، قاعدة المعرفة) تشترك في نفس
  إطار التخطيط (شريط جانبي + شريط علوي + منطقة محتوى) عبر مكوّن Layout
  واحد، فالفرق بينها هو المحتوى فقط لا الهيكل.
- حالة "أي متجر نشط الآن" مصدرها الوحيد سياق عام واحد (Store Context)،
  تقرأه كل الشاشات ولا تُمرَّر يدويًا بين المكوّنات — هذا يمنع خطأ عرض
  بيانات متجر خاطئ عند التبديل السريع، وهو امتداد مباشر لشرط العزل من
  طبقة الواجهة نفسها.

## ما بعد الاعتماد

بعد مراجعة العرض التجريبي واعتماده، الخطوة التالية حسب خارطة التنفيذ هي
تصميم الخدمات والواجهات البرمجية (API Contracts) لكل موديول، ثم البدء
الفعلي بالبناء (Backend أولًا، ثم الواجهة الأمامية الحقيقية فوق نفس نظام
التصميم هذا).
