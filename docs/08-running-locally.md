# تشغيل المنصة محليًا

**الحالة:** تم التحقق فعليًا — الخطوات أدناه هي بالضبط ما شُغِّل وتم اختباره
حيًا أثناء بناء هذا الإصدار (تسجيل دخول حقيقي، صندوق وارد حقيقي، بوابة ثقة
ذكاء اصطناعي حقيقية، تصعيد تذكرة تلقائي حقيقي).

## الطريقة الأسرع: Docker Compose + Neon

قاعدة البيانات خارجية الآن (Neon، خطة مجانية) بدل حاوية Postgres محلية —
هذا يبقيها متاحة دائمًا حتى عندما يكون جهازك مطفأ، وهو ما تحتاجه أي منصة
نشر حقيقية (Render، Fly، جهازك + Cloudflare Tunnel، ...).

1. أنشئ مشروعًا مجانيًا على [neon.tech](https://neon.tech) (إن لم يكن لديك
   واحد بالفعل)، وانسخ **connection string** الخاص بـ "Pooled connection".
2. انسخ `.env.example` إلى `.env` في جذر المشروع، والصق القيمة في
   `NEON_DATABASE_URL`.
3. شغّل:

```bash
docker compose up --build
```

الخادم الخلفي يدفع المخطط إلى Neon، يطبّق سياسات RLS، يزرع بيانات تجريبية
(6 متاجر + مستخدمون + محادثات وتذاكر حقيقية)، ثم يشغّل نفسه — كل ذلك عبر
`backend/docker-entrypoint.sh`.

- الواجهة الأمامية: http://localhost:5173
- الواجهة الخلفية: http://localhost:4000

### فتح رابط عام للعميل عبر Cloudflare Tunnel (بدون GitHub، بدون نشر سحابي)

بعد أن يعمل `docker compose up --build` محليًا:

```bash
cloudflared tunnel --url http://localhost:4000   # نافذة طرفية أولى
cloudflared tunnel --url http://localhost:5173   # نافذة طرفية ثانية
```

كل أمر يطبع رابط `https://xxxx.trycloudflare.com` عشوائيًا. ضع الرابط الأول
في `BACKEND_PUBLIC_URL` والثاني في `FRONTEND_PUBLIC_URL` داخل `.env`، ثم
أعد البناء مرة واحدة (`VITE_API_URL` يُخبز داخل حزمة الواجهة الأمامية وقت
البناء، فلا يكفي إعادة التشغيل فقط):

```bash
docker compose up --build
```

الرابط الذي تعطيه للعميل هو رابط `FRONTEND_PUBLIC_URL` (الخاص بمنفذ 5173).
ملاحظة: روابط `trycloudflare.com` المجانية غير الدائمة تتغيّر كل مرة تُعاد
فيها الأوامر — طالما نافذتا الطرفية مفتوحتان وجهازك يعمل يبقى الرابط ثابتًا؛
إغلاقهما يتطلب إعادة اللصق بروابط جديدة.

## لماذا ثلاثة أدوار في قاعدة البيانات (وليس دورًا واحدًا)

**ملاحظة عن إعداد Neon أعلاه تحديدًا:** خطة Neon المجانية تمنحك دورًا
واحدًا فقط (المالك)، ولا يمكن منحه صلاحية `BYPASSRLS` (محجوزة لمن يملك
صلاحية Superuser، وهو ما لا تملكه أي خطة مُدارة). لذلك `NEON_DATABASE_URL`
يُستخدم للثلاثة معًا هنا، وهذا يعني أن طبقة عزل RLS على مستوى قاعدة
البيانات غير فعّالة عمليًا في هذا الإعداد تحديدًا (لأن دور المالك يتجاوز
RLS افتراضيًا). العزل بين المتاجر يبقى مطبَّقًا بالكامل على مستوى الكود
(RBAC في `backend/src/middleware/rbac.ts`) — وهذا كافٍ لتجربة عميل واحد أو
عرض تجريبي، لكنه تنازل واعٍ عن طبقة الدفاع الثانية، وليس خطأ. لاستعادة
الفصل الكامل بالأدوار الثلاثة، استخدم Postgres محليًا (`docker/init.sql`)
كما هو موضّح أدناه، أو أنشئ يدويًا دورين إضافيين محدودي الصلاحيات على نفس
قاعدة بيانات Neon (ممكن عبر SQL Editor بدون حاجة لـ `BYPASSRLS`، فقط
`GRANT`s محدودة، وتترك `RESOLVER` بلا `BYPASSRLS` أيضًا إن لزم ذلك).

هذا التصميم بثلاثة أدوار مُتحقَّق منه فعليًا (وليس افتراضًا نظريًا) أثناء البناء — محاولة
استخدام دور واحد لكل شيء إما تُبطل العزل (RLS مُتجاوَز بالكامل) أو تُفشل
عمليات النظام المشروعة (استقبال webhook لا يملك سياق متجر بعد):

1. **`atlas`** (`DATABASE_URL`) — يُستخدم فقط من أدوات Prisma
   (migrate/push/seed). له `BYPASSRLS` لأن تهيئة المخطط والزرع عمليات
   إدارية تكتب عبر كل المتاجر.
2. **`atlas_app`** (`APP_DATABASE_URL`) — الدور الذي يتصل به الخادم الفعلي
   أثناء التشغيل. **بدون** `BYPASSRLS` — هذا ما يجعل سياسات
   `prisma/rls.sql` مُطبَّقة فعلاً، لا مجرد موجودة في ملف.
3. **`atlas_resolver`** (`RESOLVER_DATABASE_URL`) — يُستخدم حصرًا لتحديد "أي
   متجر يملك حساب القناة/التكامل هذا؟" عند وصول webhook عام، قبل وجود أي
   سياق متجر أصلاً. له `BYPASSRLS` لكن صلاحياته محصورة بـ `SELECT` على 4
   جداول فقط (`channel_accounts`, `channel_types`, `integrations`,
   `stores`) — وليس تجاوزًا عامًا لكل شيء.

## التشغيل اليدوي (بدون Docker)

```bash
# 1) PostgreSQL 16 + pgvector محليًا، ثم:
cd backend
cp .env.example .env   # عدّل القيم إذا لزم
npm install
npx prisma db push
npm run apply-rls
npm run seed
npm run dev             # يشغّل على :4000

# في نافذة طرفية أخرى:
cd frontend
cp .env.example .env
npm install
npm run dev             # يشغّل على :5173
```

## اختبار بوابة الثقة يدويًا (بدون قناة حقيقية)

القناة التجريبية (`mock`) تُحاكي واتساب دون أي حساب خارجي — تُستخدم بالضبط
هكذا أثناء التحقق من هذا الإصدار:

```bash
BODY='{"externalCustomerId":"c1","customerName":"عميل","text":"سياسة الاسترجاع"}'
SIG=$(node -e "const c=require('crypto');console.log(c.createHmac('sha256','dev-only-mock-secret').update(process.argv[1]).digest('hex'))" "$BODY")

curl -X POST http://localhost:4000/v1/webhooks/channels/mock/<channel-account-id> \
  -H "Content-Type: application/json" -H "x-hub-signature-256: $SIG" -d "$BODY"
```

سؤال يطابق قاعدة المعرفة المزروعة → ثقة عالية → رد آلي فوري. سؤال غير
متعلّق → ثقة منخفضة → لا رد آلي، وتُنشأ تذكرة تلقائيًا.

## متغيرات بيئة اختيارية

- `ANTHROPIC_API_KEY` — بدونه، الوكيل يُعيد نص المعرفة المطابق كما هو
  (`backend/src/lib/llm.ts`). بوجوده، يُصاغ الرد عبر Claude فعليًا دون أي
  تغيير آخر في الكود.
- أسرار توقيع القنوات/التكاملات (`WHATSAPP_APP_SECRET`, `META_APP_SECRET`,
  `SALLA_WEBHOOK_SECRET`, ...) — مطلوبة فقط لربط حسابات حقيقية؛ القناة
  والتكامل التجريبيان (`mock`) يعملان بدونها.
- `WHATSAPP_WEBHOOK_VERIFY_TOKEN` — قيمة ثابتة واحدة تختارها أنت (أي نص
  عشوائي)، تُدخَل مرة واحدة فقط في: (أ) متغيرات البيئة هنا، و(ب) إعدادات
  Webhook لتطبيق Meta نفسه (Verify Token). تُستخدم في مسار واتساب الموحّد
  `/v1/webhooks/whatsapp` الذي يخدم كل المتاجر عبر تطبيق Meta واحد — راجع
  `docs/21-meta-tech-provider-guide.md` لتفاصيل الإعداد لمرة واحدة وخطوات
  إضافة كل متجر جديد بعدها.
- `UPLOADS_DIR` — أين تُحفظ ملفات قاعدة المعرفة المرفوعة (افتراضيًا
  `backend/uploads/`؛ في Docker Compose يُشار إلى `/app/uploads` مع Volume
  مخصص `atlas_uploads` حتى تبقى الملفات بعد إعادة تشغيل الحاوية).

## رفع ملفات قاعدة المعرفة (PDF / Word / Excel)

يعمل فعليًا — لا حاجة لصيغة خاصة أو تجهيز مسبق:

```bash
curl -X POST http://localhost:4000/v1/stores/<storeId>/knowledge/sources \
  -H "Authorization: Bearer <token>" \
  -F "type=pdf" -F "title=سياسة الشحن" -F "file=@/path/to/policy.pdf"
```

يُستخرَج النص فورًا (`pdf-parse` لملفات PDF، `mammoth` لملفات Word،
`exceljs` لملفات Excel)، يُقسَّم، ويُفهرَس — دون أي خطوة يدوية إضافية. حد
الحجم 10 ميجابايت لكل ملف. نفس الأمر متاح من الواجهة عبر زر "+ إضافة مصدر
معرفة" في شاشة قاعدة المعرفة.

## ما لم يُختبر بعد

- الاتصال بحسابات واتساب/إنستغرام/ماسنجر/تيك توك أو سلة/زد/Shopify/
  WooCommerce **حقيقية** — المحوّلات (Adapters) مكتوبة بأشكال الطلبات
  الرسمية الموثّقة لكل منصة، لكنها لم تُختبر ضد حساب فعلي أو بيئة تجريبية
  (Sandbox) حقيقية، وهذا يتطلب اعتماد تطبيق فعلي من كل منصة لا يملكه أحد
  بعد. عند التنفيذ الفعلي، ابدأ بمنصة واحدة واختبرها ضد Sandbox قبل الباقي.
