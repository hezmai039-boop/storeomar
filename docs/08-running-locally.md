# تشغيل المنصة محليًا

**الحالة:** تم التحقق فعليًا — الخطوات أدناه هي بالضبط ما شُغِّل وتم اختباره
حيًا أثناء بناء هذا الإصدار (تسجيل دخول حقيقي، صندوق وارد حقيقي، بوابة ثقة
ذكاء اصطناعي حقيقية، تصعيد تذكرة تلقائي حقيقي).

## الطريقة الأسرع: Docker Compose

```bash
docker compose up --build
```

هذا وحده يكفي: يشغّل PostgreSQL (مع امتداد pgvector)، وينشئ الأدوار الثلاثة
لقاعدة البيانات تلقائيًا (`docker/init.sql`)، ثم الخادم الخلفي يدفع المخطط،
يطبّق سياسات RLS، يزرع بيانات تجريبية (6 متاجر + مستخدمون + محادثات
وتذاكر حقيقية)، ثم يشغّل نفسه — كل ذلك عبر `backend/docker-entrypoint.sh`.

- الواجهة الأمامية: http://localhost:5173
- الواجهة الخلفية: http://localhost:4000

## لماذا ثلاثة أدوار في قاعدة البيانات (وليس دورًا واحدًا)

هذا التصميم مُتحقَّق منه فعليًا (وليس افتراضًا نظريًا) أثناء البناء — محاولة
استخدام دور واحد لكل شيء إما تُبطل العزل (RLS مُتجاوَز بالكامل) أو تُفشل
عمليات النظام المشروعة (استقبال webhook لا يملك سياق متجر بعد):

1. **`atlas`** (`DATABASE_URL`) — يُستخدم فقط من أدوات Prisma
   (migrate/push/seed). له `BYPASSRLS` لأن تهيئة المخطط والزرع عمليات
   إدارية تكتب عبر كل المتاجر.
2. **`atlas_app`** (`APP_DATABASE_URL`) — الدور الذي يتصل به الخادم الفعلي
   أثناء التشغيل. **بدون** `BYPASSRLS` — هذا ما يجعل سياسات
   `prisma/rls.sql` مُطبَّقة فعلاً، لا مجرد موجودة في ملف.
3. **`atlas_resolver`** (`RESOLVER_DATABASE_URL`) — يُستخدم حصرًا لتحديد "أي
   متجر يملك حساب القناة/التكامل هذا؟" عند وصول webhook عام، قبل وجود أي
   سياق متجر أصلاً. له `BYPASSRLS` لكن صلاحياته محصورة بـ `SELECT` على 4
   جداول فقط (`channel_accounts`, `channel_types`, `integrations`,
   `stores`) — وليس تجاوزًا عامًا لكل شيء.

## التشغيل اليدوي (بدون Docker)

```bash
# 1) PostgreSQL 16 + pgvector محليًا، ثم:
cd backend
cp .env.example .env   # عدّل القيم إذا لزم
npm install
npx prisma db push
npm run apply-rls
npm run seed
npm run dev             # يشغّل على :4000

# في نافذة طرفية أخرى:
cd frontend
cp .env.example .env
npm install
npm run dev             # يشغّل على :5173
```

## اختبار بوابة الثقة يدويًا (بدون قناة حقيقية)

القناة التجريبية (`mock`) تُحاكي واتساب دون أي حساب خارجي — تُستخدم بالضبط
هكذا أثناء التحقق من هذا الإصدار:

```bash
BODY='{"externalCustomerId":"c1","customerName":"عميل","text":"سياسة الاسترجاع"}'
SIG=$(node -e "const c=require('crypto');console.log(c.createHmac('sha256','dev-only-mock-secret').update(process.argv[1]).digest('hex'))" "$BODY")

curl -X POST http://localhost:4000/v1/webhooks/channels/mock/<channel-account-id> \
  -H "Content-Type: application/json" -H "x-hub-signature-256: $SIG" -d "$BODY"
```

سؤال يطابق قاعدة المعرفة المزروعة → ثقة عالية → رد آلي فوري. سؤال غير
متعلّق → ثقة منخفضة → لا رد آلي، وتُنشأ تذكرة تلقائيًا.

## متغيرات بيئة اختيارية

- `ANTHROPIC_API_KEY` — بدونه، الوكيل يُعيد نص المعرفة المطابق كما هو
  (`backend/src/lib/llm.ts`). بوجوده، يُصاغ الرد عبر Claude فعليًا دون أي
  تغيير آخر في الكود.
- أسرار توقيع القنوات/التكاملات (`WHATSAPP_APP_SECRET`, `META_APP_SECRET`,
  `SALLA_WEBHOOK_SECRET`, ...) — مطلوبة فقط لربط حسابات حقيقية؛ القناة
  والتكامل التجريبيان (`mock`) يعملان بدونها.

## ما لم يُختبر بعد

- الاتصال بحسابات واتساب/إنستغرام/ماسنجر/تيك توك أو سلة/زد/Shopify/
  WooCommerce **حقيقية** — المحوّلات (Adapters) مكتوبة بأشكال الطلبات
  الرسمية الموثّقة لكل منصة، لكنها لم تُختبر ضد حساب فعلي أو بيئة تجريبية
  (Sandbox) حقيقية، وهذا يتطلب اعتماد تطبيق فعلي من كل منصة لا يملكه أحد
  بعد. عند التنفيذ الفعلي، ابدأ بمنصة واحدة واختبرها ضد Sandbox قبل الباقي.
